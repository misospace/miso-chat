## 1. Current State Assessment

### 1.1 Codebase Structure (as of current main)

```
lib/
  gateway-ws.js      — Gateway WebSocket manager (OpenClaw-specific)
  db.js              — SQLite/database layer
  emoji-shortcodes.js — Emoji shortcode mapping
  reaction-events-*.js — Reaction event handling (browser/node variants)
  render-utils.js    — UI rendering helpers
  ssrf-validation.js — SSRF protection
  update-manager.js  — Self-update logic

server.js            — Express server, API routes, session management
```

### 1.2 OpenClaw Hardcoding Found

| Location | What's hardcoded | Risk level |
|---|---|---|
| `lib/gateway-ws.js:15` | WS URL: `ws://openclaw.llm.svc.cluster.local:18789` | **High** — cluster-internal endpoint, not abstracted |
| `server.js:992-995` | Gateway URL env vars default to OpenClaw paths | **High** — no multi-provider fallback |
| `server.js:1127,1158,1233` | `provider: 'openclaw'` hardcoded in session/message records | **Medium** — works if single-provider, blocks multi-provider |
| `server.js:1484-1511` | `/api/openclaw-status`, `/api/openclaw-stop` endpoints | **Low-Medium** — provider-named routes |
| `server.js:891` | Device identity path under `~/.openclaw/identity/` | **Low** — OpenClaw-specific identity store |

### 1.3 What's Already Capability-Driven

- **Gateway WS protocol:** Uses frame-based `type`/`event`/`payload` structure with `connect.challenge`, `request`, `response`, `event` patterns — this is already somewhat abstracted from provider specifics
- **Session model:** Sessions have IDs, messages, and a provider field (currently always 'openclaw')

### 1.4 What's Not Yet Capability-Driven

- No capability discovery mechanism (UI doesn't query what the backend can do)
- No normalized event contract — events are OpenClaw-specific WS frames
- Timeline rendering is not decoupled from provider assumptions
- Auth/session ownership is OpenClaw-centric

## 2. Proposed Capability Contract (Draft)

Based on the issue's guidance, here's a proposed capability schema:

```typescript
interface SessionCapabilities {
  // Message operations
  can_reply: boolean;
  can_edit: boolean;
  can_delete: boolean;
  can_quote_reply: boolean;
  
  // Rich interactions
  can_react: boolean;
  can_send_attachments: boolean;
  can_send_voice: boolean;
  
  // Conversation structure
  can_thread: boolean;
  can_abort: boolean;
  
  // Presence & status
  supports_presence: boolean;
  supports_typing_indicator: boolean;
  
  // Delivery guarantees
  supports_delivery_receipts: boolean;
  supports_read_receipts: boolean;
  
  // Runtime features
  supports_tools: boolean;
  supports_approvals: boolean;
  supports_agent_identity: boolean;
}

interface MessageEnvelope {
  id: string;
  session_id: string;
  role: 'user' | 'assistant' | 'system' | 'runtime';
  content: string;
  attachments?: Attachment[];
  capabilities_used: Capability[];
  provider_metadata?: Record<string, unknown>; // non-authoritative
  created_at: number;
  updated_at?: number;
}

interface TimelineEvent {
  type: 'message.created' | 'message.updated' | 'reaction.added' | 
        'reaction.removed' | 'typing.started' | 'typing.stopped' |
        'presence.online' | 'presence.offline' | 'delivery.read' |
        'attachment.available' | 'run.started' | 'run.completed' | 
        'run.failed' | 'tool.approval_required';
  payload: Record<string, unknown>;
  timestamp: number;
}
```

## 3. Migration Strategy (Incremental)

### Phase 1: Capability Discovery (Foundation)
- [ ] Add `GET /api/capabilities` endpoint that queries the backend for supported features
- [ ] Cache capabilities per-session or globally
- [ ] UI reads capabilities before rendering feature buttons

### Phase 2: Event Contract (Timeline Decoupling)
- [ ] Define normalized event envelope format
- [ ] Create event adapter layer: OpenClaw WS frames → normalized events → other providers in future
- [ ] Timeline component subscribes to normalized events, not raw provider frames

### Phase 3: Provider Metadata Layer (Non-Authoritative)
- [ ] Keep `provider` field in messages but make it metadata-only
- [ ] Remove provider-based branching from core UI logic
- [ ] Add provider badges as optional decorations

### Phase 4: Auth/Session Boundaries
- [ ] Explicit session authorization checks
- [ ] Channel/session access control audit
- [ ] Provider account binding review

## 4. Anti-Goals (Reaffirmed from Issue)

- ❌ Do not build fully custom UI behavior per provider
- ❌ Do not overfit to one MCP implementation detail
- ❌ Do not assume feature parity across providers
- ❌ Do not move product logic into provider adapters if OpenClaw can normalize upstream

## 5. Open Questions (From Issue)

1. **What capability/event surfaces should be guaranteed by OpenClaw vs optional?**
   - *Initial assessment:* `can_reply`, `can_edit`, `can_abort`, `supports_delivery_receipts` should be guaranteed. Others like `can_send_voice`, `supports_presence` are provider-dependent.

2. **What minimal runtime state does miso-chat need for a good operator experience?**
   - *Initial assessment:* Tool status (running/completed/failed), approval pending, session active/idle, agent identity — these drive the UI's "runtime affordances" section.

3. **Which existing UI paths branch on backend/provider assumptions?**
   - *Initial assessment:* The `/api/openclaw-status` and `/api/openclaw-stop` endpoints are the clearest examples. The WS connection initialization in `gateway-ws.js` also assumes OpenClaw protocol.

4. **How should attachments, replies, and system events be unified?**
   - *Initial assessment:* Attachments as first-class message properties with capability gate. Replies via `can_quote_reply`. System events as `role: 'runtime'` messages in the timeline.
